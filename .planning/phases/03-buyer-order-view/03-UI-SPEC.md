## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`gap-1`, `p-1`) | Icon gaps, tag spacing |
| sm | 8px (`gap-2`, `p-2`) | Compact element spacing, badge padding |
| md | 16px (`gap-4`, `px-4`) | Default horizontal page padding |
| lg | 24px (`p-6`) | Section card padding |
| xl | 32px (`py-8`) | Layout vertical gaps |
| 2xl | 48px (`py-12`) | Empty/loading state vertical centering |
| 3xl | 64px (`py-16`) | EmptyState top/bottom padding |

Exceptions:
- Timeline dot: 12px diameter (`w-3 h-3`) — not on scale, chosen for visual weight at 14px body text
- Timeline vertical line: 2px width (`w-0.5`) — thin connector, not a spacing token
- Header height: 56px (`h-14`) — existing fixed value, not modified
- Touch targets (Cancel button): minimum 44px height (`py-3` on full-width button ≈ 45px (meets 44px minimum))
- `mt-0.5` / `mb-0.5` (2px): micro-nudge for optical alignment of stacked `text-xs` lines within tight card headers — keeps labels visually flush without adding a full 4px gap (used on date and order-number lines in status card, and on rejection block title)
- `-mt-0.5` (−2px): negative offset to counteract Nunito's built-in line-height descender gap in timeline dot vertical alignment

### Status Color Mapping (Timeline Dots + Rejection Block)

Derived from `OrderStatusBadge` in `Badge.tsx` — same color semantics, applied to timeline dots.

| Status | Dot Background | Dot Ring | Label Color | Badge Variant |
|--------|---------------|----------|-------------|---------------|
| `pending` | `bg-yellow-400` | `ring-yellow-200` | `text-yellow-700` | warning |
| `confirmed` | `bg-blue-400` | `ring-blue-200` | `text-blue-700` | info |

**Cancel button (conditional — `status === 'pending'` only):**
```
<button
  onClick={handleCancel}
  disabled={cancelling}
  className="w-full border-2 border-red-300 text-red-600 font-bold py-3 rounded-2xl
             flex items-center justify-center gap-2 active:scale-95 transition-transform
             disabled:opacity-50 disabled:cursor-not-allowed"

      Cancelar pedido
    </>
  )}
</button>
```

Design rationale: outlined (border-only) rather than filled red — destructive but not alarming. Single-tap, no dialog (D-06). Full-width for easy mobile tap target (min ≈45px height via `py-3`).

**Loading state (initial fetch):**
```
<PageLoader />   ← existing component, same pattern as OrderHistory
```

No spinner flash on 15s polling refresh: update state with `setOrder(data)` silently. Never show `PageLoader` again after initial load; only show it while `order === null && loading === true`.

**Empty / not-found state:**
```
<EmptyState
  title="Pedido não encontrado"
  description="Este pedido não existe ou você não tem permissão para visualizá-lo."
  icon={<PackageOpen size={64} />}
  action={
    <button onClick={() => navigate('/orders')} className="...">
      Ver meus pedidos
    </button>
  }
/>
```

**Header back behavior:** Use existing `<Header showBack />` — calls `navigate(-1)` (history back). This is correct for push deep-link entry (browser history has entry to go back to) and for navigation from confirmation overlay's "Ver Pedido" button.

**Scroll behavior:** `window.scrollTo(0, 0)` in `useEffect([], [])` on mount — prevents inheriting scroll position from Cart overlay.

## Interaction Patterns and State Transitions

### OrderDetail Page — Full State Machine

| UI State | Trigger | Visual |
|----------|---------|--------|
| Loading | Initial mount, `order === null` | `<PageLoader />` full screen |
| Loaded | `getOrderById` returns Order | Full page content |
| Polling | Every 15s silently | No visual change; badge/timeline update if status changed |
| Cancelling | User taps "Cancelar pedido" | Button disabled, spinner in button; rest of page unchanged |
| Cancelled | PATCH succeeds | Order reload; cancel button disappears; timeline gains new entry; `toast.success("Pedido cancelado")` |
| Cancel error | PATCH fails | `toast.error("Erro ao cancelar pedido. Tente novamente.")` ; button re-enabled |
| Not found | `getOrderById` returns null | `<EmptyState>` with navigate-back button |
| Load error | `getOrderById` throws | `toast.error("Erro ao carregar pedido")` + EmptyState with retry button |

### Timeline Construction Logic

```
timelineEntries =
  (status_history?.length > 0)
    ? [...status_history].reverse()          // newest first
    : [{ status: order.status, at: order.updated_at ?? order.created_at }]
```

Each entry renders: colored dot + PT-BR status label + formatted timestamp.
No future placeholders — only actual history.

### Confirmation Overlay — Interaction States

| Element | State | Behavior |
|---------|-------|---------|
| WhatsApp CTA | Always enabled | Opens WhatsApp; sets `whatsappOpened = true`; marks order sent |
| "Ver Pedido" button | Always enabled | Clears `checkoutSuccess`; navigates to `/orders/${orderId}` |
| Scroll | Content overflow | `flex-1 overflow-y-auto` on content area; buttons always visible |

**WhatsApp gate removed from "Ver Pedido":** The original "Ver meus pedidos" button was disabled until WhatsApp was opened — this gate was punitive. "Ver Pedido" is always tappable. The user can always check order status without sending WhatsApp.

## Accessibility and Responsive Behavior

### Responsive

- **Container:** `max-w-lg mx-auto` inherited from BuyerLayout — all content stays within 512px column
- **Minimum touch targets:** all interactive elements minimum 44px height (`py-3` ≈ 45px actual, meets 44px minimum)
- **Scroll:** OrderDetail page scrolls normally; Header sticky at top; bottom nav clearance via `pb-24`
- **Overlay scroll:** confirmation overlay uses `flex-1 overflow-y-auto` on content — action buttons always pinned to bottom

### Accessibility

- **Cancel button:** `aria-label` is button text ("Cancelar pedido") — sufficient; no separate aria needed
- **Timeline:** rendered as a `<div>` list, not `<ul>` — acceptable for this context; each entry is non-interactive (no role needed)
- **Header back button:** existing `aria-label="Voltar"` on ArrowLeft button — unchanged
- **Loading spinner:** no `aria-busy` required for polling (silent update); initial `<PageLoader />` uses existing component
- **Rejection block:** `AlertTriangle` icon is decorative alongside text — no `aria-label` on icon; text carries meaning
- **Color contrast:** all text-on-background combinations satisfy WCAG AA:
  - `text-gray-700` on `bg-white`: ratio ≥ 4.6:1
  - `text-primary` (`#2d6a4f`) on `bg-white`: ratio ≥ 4.5:1
  - `text-red-600` on `bg-red-50`: ratio ≥ 4.5:1
  - `text-yellow-700` on `bg-white`: ratio ≥ 4.6:1
  - White text on `bg-green-500` (WhatsApp button): ratio ≥ 4.5:1

### iOS PWA specifics

- **Safe area bottom:** `pb-24` on scroll container provides clearance for both BuyerNav and iOS home indicator
- **Tap highlight:** `active:scale-95 transition-transform` on buttons — provides haptic-aligned tap feedback without browser default highlight
- **No pinch zoom issues:** all font sizes ≥ 12px; inputs not used on these screens
