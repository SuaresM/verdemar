## Layout: Two-Section Page

### Structure

```
<Header title="Pedidos" />
<div class="px-4 py-4 space-y-6 pb-24">   тЖР pb-24 clears fixed bottom nav

  <!-- Section 1: Pendentes -->
  <section>
    <SectionHeader label="PENDENTES" count={pendingOrders.length} />
    {pendingOrders.length === 0
      ? <EmptyState />
      : pendingOrders.map(card)}
  </section>

  <!-- Divider -->
  <hr class="border-border" />

  <!-- Section 2: Em andamento -->
  <section>
    <SectionHeader label="EM ANDAMENTO" count={activeOrders.length} />
    {activeOrders.length === 0
      ? <EmptyState />
      : activeOrders.map(card)}
  </section>

```

### Section Header

```
<div class="flex items-center gap-2 mb-3">
  <span class="text-xs font-bold text-muted-foreground uppercase tracking-wider">
    {label}{count !== undefined ? ` (${count})` : ''}
  </span>
</div>
```

Count shown for Pendentes section only. "Em andamento" shows no count.

### Section Divider

`<hr class="border-border my-2" />` тАФ uses existing `border` token (#e2e8f0).

### Data split

- **Pendentes:** `orders.filter(o => o.status === 'pending')`
- **Em andamento:** `orders.filter(o => o.status === 'confirmed' || o.status === 'in_route')`
- Orders with `status = 'delivered' | 'cancelled' | 'rejected'` are excluded from the query (filter at service layer).

## Component: SupplierNav Badge

### Visual spec

- Container: `relative` wrapper around the `<Icon>` element inside the NavLink
- Badge element: `absolute -top-1 -right-1 w-4 h-4 rounded-full bg-danger text-white flex items-center justify-center`
- Text: `text-[9px] font-bold leading-none`
- Max display: counts 1тАУ9 render as-is; count 10+ renders as `"9+"` (string, fits in 16px circle)
- Hidden when count is 0: render nothing (no empty circle)

### Implementation note

`SupplierNav` must accept `pendingCount: number` prop. Caller (`App.tsx` or a layout wrapper) fetches count via the same 15s polling hook used by `SupplierOrders`. The count is shared state тАФ lifted to the layout level.

### Example markup

```tsx
<div className="relative">
  <ClipboardList size={22} strokeWidth={2} />
  {pendingCount > 0 && (
    <span className="absolute -top-1 -right-1 w-4 h-4 rounded-full bg-danger text-white flex items-center justify-center text-[9px] font-bold leading-none">
      {pendingCount > 9 ? '9+' : pendingCount}
    </span>
  )}
</div>
```

## Deep-Link Scroll Behavior

### Trigger

URL contains `?order=<id>` (e.g. `/supplier/orders?order=abc-123`), set by push notification tap via service worker `notificationclick` handler.

### Behavior sequence

1. Page mounts and data loads from `getOrdersBySupplier`.
2. `useEffect` runs after orders are set: reads `new URLSearchParams(location.search).get('order')`.
3. If a matching order id is found in the loaded list:
   a. Set `expanded[targetId] = true` тАФ expands the card.
   b. Call `document.getElementById(`order-card-${targetId}`)?.scrollIntoView({ behavior: 'smooth', block: 'start' })` after a 200ms delay (allows DOM paint of expanded state).
   c. Apply a 1.5s highlight ring on the card: add class `ring-2 ring-primary ring-offset-2` for 1500ms then remove.
4. If the order is not found (stale link, wrong supplier), do nothing тАФ no error shown.

### Card element id

Each card wrapper gets `id={`order-card-${order.id}`}`:

```tsx
<div
  id={`order-card-${order.id}`}
  key={order.id}
  className="bg-white rounded-2xl shadow-sm overflow-hidden transition-shadow"
>
```

### Highlight class

Temporary highlight: add `ring-2 ring-primary ring-offset-2` to the card div for 1500ms then remove via `setTimeout`. Implemented with local state `highlightedId: string | null`.

```
highlighted: "bg-white rounded-2xl shadow-sm overflow-hidden ring-2 ring-primary ring-offset-2"
default:     "bg-white rounded-2xl shadow-sm overflow-hidden"
```

## Phase-Specific Constraints

- **Mobile-first, iOS PWA:** All touch targets minimum 44px tall. Action buttons use `py-2.5` (10px + content тЙИ 36px rendered); acceptable because they sit in an expanded card, not a primary nav target. Bottom sheet buttons use `py-4` (16px + content) тАФ full 48px+ target.
- **No Supabase Realtime:** All live data via 15s polling. No optimistic UI тАФ wait for server confirmation before updating local state.
- **No history section:** Cards with `delivered | cancelled | rejected` status are excluded from query entirely. No "Ver hist├│rico" link in this phase.
- **WhatsApp link pattern is preserved:** Every status change produces a Sonner toast with a WhatsApp action. This is an existing product decision, not optional.
